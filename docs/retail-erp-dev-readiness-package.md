# 零售 ERP 开发就绪包

## 1. 当前结论

基于现有需求文档 `ERP系统需求文档.md`，项目已经具备立项、技术选型、架构设计、模块拆分的基础，但还不具备“大规模并行开发”的条件。

当前建议进入的是 **Phase 0：开发准备阶段**，而不是直接进入所有业务域的全面开发。

### 1.1 可以立即启动的工作
- 技术架构选型与仓库初始化
- 权限、审计、组织、门店、仓库、商品等基础域建模
- API 规范、数据字典、状态机、集成协议设计
- 核心链路的原型补齐与业务规则确认

### 1.2 暂不建议直接全面启动的工作
- 高复杂度促销叠加引擎
- 多平台全量订单履约策略
- 自动凭证全量自动化
- 智能补货与高级分析模型

## 2. 建议的 V1 范围切片

现有需求文档中的 MVP 范围偏大。若希望在 10 至 12 周内完成首个可上线版本，建议将 V1 切成如下范围。

### 2.1 V1.0 必须纳入

| 业务域 | 纳入范围 | 约束说明 |
|---|---|---|
| 组织与权限 | 组织、区域、门店、用户、角色、数据权限、审计日志 | 仅支持总部/区域/门店三级数据隔离 |
| 商品中心 | SPU/SKU、条码、分类、品牌、规格、上下架、基础价格 | 价格仅支持“基础售价 + 门店覆盖价”，暂不做复杂时段价 |
| 库存中心 | 可售/锁定/在途/残次库存，入库、出库、调拨、盘点、损耗 | 先做单仓账实一致，再支持仓到店/店到店 |
| 门店 POS | 收银、退货、换货、会员识别、交接班、日结 | 先支持在线模式；离线能力单列待确认 |
| 全渠道订单 | 对接 1 个电商平台；订单同步、库存占用、发货、售后退货 | 先固定 1 个平台，避免多平台口径差异 |
| 财务结算 | 门店日结对账、应收应付基础台账、人工审核凭证生成 | 自动记账先做“生成待审核凭证”，不直接自动过账 |
| 报表 | 门店销售日报、库存日报、日结对账报表 | 先做经营看板基础版，不做自助分析 |

### 2.2 V1.1 再纳入

| 业务域 | 建议后移能力 | 后移原因 |
|---|---|---|
| 促销 | 多规则叠加、赠品链路、组合促销引擎 | 规则复杂，开发和测试成本高 |
| 会员 | 标签、成长值、复杂积分规则、活动效果归因 | 依赖交易口径先稳定 |
| 补货 | 智能补货建议、销售预测、补货策略自动化 | 依赖历史销量与库存数据质量 |
| 财务 | 自动凭证过账、费用分摊、区域利润核算 | 依赖会计科目与核算规则细化 |
| 分析 | 会员复购、同店增长、坪效、人效深度模型 | 依赖 BI 口径统一 |

### 2.3 V1.0 上线门槛
- 单店 POS 主流程稳定可用
- 仓到店调拨和电商订单占库不冲突
- 库存准确率、日结成功率和关键交易成功率可被监控
- 所有 P0 单据具备审计能力

## 3. 开发前必须补齐的文档

以下内容补齐后，研发可以进入真正的并行开发。

| 编号 | 文档/产物 | 当前状态 | 责任方 | 完成标准 |
|---|---|---|---|---|
| R1 | 业务规则说明 | 缺失 | 产品 + 业务 | 关键链路规则均有优先级与异常口径 |
| R2 | 数据字典 | 缺失 | 产品 + 架构 + 后端 | 核心实体具备字段、类型、约束、状态说明 |
| R3 | 状态机定义 | 缺失 | 产品 + 后端 | 订单、库存、售后、日结状态可枚举 |
| R4 | 页面原型 | 缺失 | 产品/设计 | POS、订单、库存、日结页面均具备高保真或线框图 |
| R5 | 外部接口规格 | 缺失 | 架构 + 对接方 | 明确平台、协议、鉴权、重试、回调口径 |
| R6 | UAT 场景 | 缺失 | 产品 + 测试 | 每个 P0 功能至少 1 个主流程和 2 个异常流程 |
| R7 | 主数据迁移方案 | 缺失 | 实施 + 数据 | 商品、门店、库存、会员来源和清洗规则明确 |

## 4. 核心业务规则清单

以下规则不明确，后端实现、前端交互和测试用例都会反复返工。建议逐条确认。

### 4.1 价格规则
- 基础售价、门店覆盖价、活动价三者优先级如何排序
- 是否允许同一 SKU 在同一时段存在多个有效价格
- 售价变更是否影响已创建未支付订单
- 门店手工改价是否允许，允许时需要哪些权限和日志

### 4.2 促销规则
- 是否允许满减和优惠券叠加
- 折扣和赠品是否允许叠加
- 同一商品命中多个活动时按“最低价”还是“优先级最高活动”处理
- 整单级优惠在退货时如何分摊到商品行
- 促销失败时收银台是否允许人工兜底

### 4.3 库存规则
- 可售、锁定、在途、残次库存是否允许相互直接转换
- 电商订单占库的生效时点：下单、支付、审核还是配货
- 未支付订单的占库超时时间
- 门店 POS 销售扣减库存的时点：支付成功前还是成功后
- 盘点差异单是否允许直接生效，还是必须审批

### 4.4 售后与退换货规则
- 无单退货的门店权限范围和金额上限
- 原单退货是否必须校验原支付方式
- 换货是“退后再买”还是单据级一体化处理
- 电商退货回仓后，如何判定可二次销售、残次、报损

### 4.5 日结与对账规则
- 交接班的现金差异是否允许直接提交
- 日结的口径是否包含未结算第三方支付
- 银行/支付渠道回单延迟时，是否允许 T+1 对账补录
- 对账差异的责任流转规则和关闭规则

### 4.6 财务与凭证规则
- 自动生成凭证的触发时点：交易完成、日结完成还是月结完成
- 凭证是否允许重复生成，若允许如何冲销
- 门店费用是否在 V1.0 纳入利润核算
- 退款、退货、红冲的会计处理规则是否统一

## 5. 数据字典模板

### 5.1 字段模板

| 字段项 | 说明 |
|---|---|
| 实体名称 | 例如 `sku`、`inventory_balance`、`pos_order` |
| 字段名 | 英文字段名，统一 snake_case |
| 中文名 | 中文业务含义 |
| 类型 | `varchar`、`bigint`、`numeric(18,2)`、`timestamp` 等 |
| 必填 | 是 / 否 |
| 默认值 | 若有则填写 |
| 唯一约束 | 是否唯一，唯一范围是什么 |
| 枚举值 | 若为状态字段，列出全部枚举 |
| 来源 | 人工录入 / 系统生成 / 外部同步 |
| 业务规则 | 字段的校验与计算规则 |
| 示例值 | 样例数据 |

### 5.2 核心实体最小清单

| 实体 | 目的 | 最小必备字段 |
|---|---|---|
| `org_unit` | 组织与区域管理 | `id` `code` `name` `type` `parent_id` `status` |
| `store` | 门店管理 | `id` `code` `name` `region_id` `status` `open_date` |
| `warehouse` | 仓库管理 | `id` `code` `name` `type` `store_id` `status` |
| `sku` | 商品库存单位 | `id` `sku_code` `barcode` `name` `brand_id` `category_id` `status` |
| `sku_price` | 商品价格 | `id` `sku_id` `store_id` `price_type` `sale_price` `effective_from` `effective_to` |
| `inventory_balance` | 库存余额 | `id` `owner_type` `owner_id` `sku_id` `stock_status` `qty` `updated_at` |
| `inventory_txn` | 库存流水 | `id` `biz_type` `biz_no` `sku_id` `from_status` `to_status` `qty` `occurred_at` |
| `purchase_order` | 采购订单 | `id` `order_no` `supplier_id` `warehouse_id` `status` `total_amount` |
| `pos_order` | POS 小票主单 | `id` `order_no` `store_id` `cashier_id` `member_id` `status` `payable_amount` `paid_amount` |
| `pos_order_item` | POS 小票明细 | `id` `order_id` `sku_id` `qty` `list_price` `discount_amount` `net_amount` |
| `omni_order` | 电商订单 | `id` `platform` `platform_order_no` `store_id` `fulfillment_node_id` `status` |
| `return_order` | 退货单 | `id` `return_no` `source_order_type` `source_order_id` `reason_code` `status` |
| `member` | 会员档案 | `id` `member_no` `mobile` `name` `level` `status` |
| `coupon_account` | 券账户 | `id` `member_id` `coupon_id` `status` `issued_at` `used_at` |
| `store_day_close` | 门店日结 | `id` `store_id` `biz_date` `shift_no` `cash_amount` `third_pay_amount` `status` |
| `voucher_stub` | 待审核凭证 | `id` `source_biz_type` `source_biz_no` `accounting_date` `status` |
| `audit_log` | 审计日志 | `id` `operator_id` `operator_name` `biz_type` `biz_id` `action` `before_json` `after_json` `created_at` |

### 5.3 通用技术字段

所有核心业务表建议默认包含：
- `id`
- `created_at`
- `created_by`
- `updated_at`
- `updated_by`
- `version`
- `is_deleted`

## 7. 建议的迭代计划

### 7.1 Phase 0：开发准备（2 周）
- 确认 V1.0 范围边界
- 输出业务规则说明
- 输出数据字典和状态机
- 确认 1 个电商平台、1 套支付渠道、1 套财务接口
- 完成仓库初始化、CI、环境划分和基础编码规范

### 7.2 Sprint 1：基础域（2 周）
- 完成组织、门店、仓库、角色权限
- 完成商品、分类、条码、基础价格
- 完成审计日志基础能力

### 7.3 Sprint 2：库存域（2 周）
- 完成库存余额、库存流水、入库/出库/调拨
- 完成盘点与损耗基础流程
- 打通商品和库存数据关系

### 7.4 Sprint 3：POS 域（2 周）
- 完成 POS 收银、支付结果落单、退货、换货
- 完成交接班和日结主流程
- 打通 POS 与库存、会员的读写链路

### 7.5 Sprint 4：全渠道订单（2 周）
- 对接 1 个电商平台
- 实现订单同步、占库、配货、发货、售后退货
- 完成库存占用与释放规则

### 7.6 Sprint 5：财务与报表（2 周）
- 完成门店日结对账
- 生成待审核凭证
- 输出销售日报、库存日报、日结报表

### 7.7 Sprint 6：联调与试点（2 周）
- 完成门店试点联调
- 修复高频异常场景
- 完成 UAT 与上线清单

## 8. 开发启动前的准入条件

以下条件全部满足，才建议进入并行开发：
- V1.0 范围冻结
- 业务规则文档完成评审
- 数据字典完成评审
- 外部接口规格冻结
- 页面原型完成
- UAT 场景清单已定义
- 环境、日志、监控、发布流程已建立
